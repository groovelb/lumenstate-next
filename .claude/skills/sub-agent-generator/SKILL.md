# Subagent Generator

사용자의 한 줄짜리 의도를 well-formed subagent 정의로 변환하는 스킬이다. description 작성 규칙, 도구 최소권한 매트릭스, 본문 스켈레톤을 따라 일관된 품질의 subagent를 만든다.

## 워크플로우

### 1단계: 의도 파악

사용자가 다음 항목을 모두 명시했는지 확인한다.

- 주요 작업(이 에이전트가 무엇을 하는가)
- 트리거 조건(언제 발동되어야 하는가)
- 읽기 전용인가, 쓰기 권한이 필요한가
- 프로젝트 레벨인가, 유저 레벨인가
- 모델 선호도(기본값 sonnet)

빠진 항목이 있으면 한 번에 묻되 최대 3개까지만. 그래도 모호하면 기본값으로 채우고 리포트에 명시한다.

기본값 fallback:
- 트리거: "사용자가 명시 호출할 때만"
- 권한: "읽기 전용"
- 위치: "프로젝트 레벨(`.claude/agents/`)"
- 모델: sonnet

### 2단계: reference.md 정독

이 스킬 폴더의 `reference.md`에 description 작성 규칙, 도구 매트릭스, 모델 선택 휴리스틱, 본문 스켈레톤이 들어있다. 파일을 쓰기 전에 반드시 읽고 그 룰을 따른다.

### 3단계: 템플릿 선택

에이전트의 모양에 따라 `templates/`에서 시작점을 고른다.

- `reviewer.md`: 읽기 전용 검사(리뷰, 감사, 분석)
- `mutator.md`: 코드/파일 수정(리팩터, 포맷, 마이그레이션, 테스트 실행)
- `analyzer.md`: 다단계 추론(디버그, 근본 원인, 아키텍처 검토)

세 가지 모두에 안 맞으면 reference.md의 본문 스켈레톤만 들고 처음부터 작성한다.

### 4단계: 파일 생성

YAML frontmatter와 본문을 다음 규칙으로 작성한다.

**name 필드**
소문자, 하이픈 구분, 역할 명사로 짓는다. `migration-writer`(O), `write-migration`(X). Claude가 이 이름으로 호출하므로 의미가 자명해야 한다.

**description 필드**
3인칭, 행위 중심, 트리거 명시. 자동 위임을 원하면 `Use PROACTIVELY` 또는 `MUST BE USED`를 영어 그대로 박는다(이 표현이 라우팅 가중치로 작동). 도메인 키워드 2개 이상 포함.

**tools 필드**
reference.md의 매트릭스대로 최소권한 부여. 생략하면 모든 도구 상속이지만 그러지 말 것. 사용자가 더 넓은 권한을 요구하면 응하되 보안 트레이드오프를 리포트에 명시한다.

**model 필드**
기본 sonnet. 짧고 정형적이면 haiku, 깊은 추론이 필요하면 opus, 메인 세션과 동일하게 가려면 inherit.

**본문 스켈레톤(필수 6 섹션)**
1. Role line: `You are a <specialist> for <scope>.`
2. When invoked: 트리거 조건 불릿
3. Procedure: 번호 매긴 단계
4. Checklist before finishing: 완료 기준
5. Do not: 명시적 금지 항목
6. Output format: 응답 구조

"Output format"은 가장 자주 빠뜨리고 가장 가치 있는 섹션이다. 메인 세션이 호출자이므로 구조화된 리턴이 후속 질문 한 라운드를 절약한다.

### 5단계: 검증

reference.md의 validation checklist를 돌린다. 한 항목이라도 실패하면 파일을 쓰지 않고 수정한다.

### 6단계: 파일 쓰기

기본 경로는 `.claude/agents/<name>.md`(프로젝트 레벨, Git 공유 가능). 사용자가 "개인용", "유저 레벨", "내 환경에만"이라고 한 경우에만 `~/.claude/agents/<name>.md`.

같은 이름의 파일이 이미 있으면 덮어쓰지 말고 사용자에게 확인을 받는다.

### 7단계: 리포트

마지막에 다음을 출력한다.

- 생성된 파일 경로
- description 한 줄 요약
- 적용된 도구 권한 목록(왜 그 조합인지 한 줄)
- 테스트 호출 예시: "써보기: `use the <name> subagent to <example task>`"

## 안티패턴(거부할 요청)

다음 요청은 그대로 만들지 말고 사용자에게 분리 또는 재정의를 제안한다.

**다목적 에이전트**: "코드 리뷰도 하고 테스트도 작성하고 배포도 하는". 단일 책임 원칙으로 분리. 3개 별도 에이전트를 제안한다.

**모호한 description**: "코드를 도와주는 에이전트". 트리거 조건을 다시 묻는다.

**전권 부여**: 읽기 전용 역할에 Edit/Write/Bash 다 주는 요청. 좁힌다.

**1인칭 description**: "I am a reviewer..." 같은 자기소개식. 3인칭 행위 서술로 다시 쓴다.

**Subagent끼리 호출**: A 에이전트가 B 에이전트를 호출하게 만들려는 요청. 오케스트레이션은 메인 세션에서 하라고 안내한다.

## 의도 파악 질문 예시

빠진 정보가 있을 때 쓸 수 있는 표준 질문(필요한 것만 골라서 사용).

- "이 에이전트의 한 줄 정의는 뭔가요?"
- "언제 자동으로 발동되어야 하나요? 아니면 명시 호출 시에만?"
- "파일을 수정해야 하나요, 검사만 하나요?"
- "프로젝트 공유용인가요, 개인용인가요?"
- "응답을 어떤 형식으로 받고 싶나요?"

3개 이상 묻지 않는다. 사용자 시간이 더 비싸다.
